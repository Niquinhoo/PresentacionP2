Arquitectura del Sistema

El sistema sigue una arquitectura multicapa estrictamente separada. Cada capa tiene una única responsabilidad y se comunica con la capa adyacente mediante interfaces bien definidas, favoreciendo la mantenibilidad y los tests unitarios.

🖼️ Capa Vista (GUI)

Formularios Java Swing construidos con NetBeans Form Builder. Responsable de la interacción con el usuario. Utiliza SwingWorker para operaciones asíncronas y nunca bloquea el EDT.

Login.java Menu.java MesasPanel ReportesPanel CheckoutDialog

⚙️ Capa de Servicio / Controlador

Contiene la lógica de negocio. Los Controllers actúan como puente entre GUI y Services. Los Services orquestan las operaciones del dominio: validaciones, transacciones, reglas de negocio. ServicioFactory implementa Singleton con double-checked locking.

MesaService PedidoService ProductoService ReporteService ServicioFactory

🗄️ Capa DAO (Data Access Object)

Abstrae el acceso a datos. Cada entidad tiene una interfaz (contrato) y una implementación concreta con JDBC/SQL. Permite swappear la fuente de datos sin tocar la capa de servicio. Las operaciones críticas (como insertar pedido + detalles) son transaccionales.

MesaDAO PedidoDAO ProductoDAO UsuarioDAO CategoriaDAO ReporteDAO

📦 Capa Modelo

POJOs que representan las entidades del dominio. No tienen lógica de negocio. Utilizan enumeraciones para los estados (EstadoMesa, EstadoPedido, Rol), garantizando type-safety en todo el sistema.

Mesa Pedido Producto Usuario Categoria DetallePedido

🔌 Capa de Persistencia

Base de datos MySQL alojada en TiDB Cloud. La conexión se gestiona a través de ConexionDB (Singleton) con pool HikariCP. Configuración externalizada en db.properties.

ConexionDB HikariCP MySQL/TiDB db.properties

Capa DAO — Contratos

Cada entidad expone una interfaz DAO con operaciones CRUD estándar. Las implementaciones usan JDBC puro con PreparedStatement para prevenir inyección SQL.

MesaDAO

CRUD completo + búsqueda por estado. Permite filtrar mesas LIBRES para asignación rápida. Método clave: buscarPorEstado(EstadoMesa).

PedidoDAO

Operaciones transaccionales para INSERT simultáneo de pedido + detalle_pedido. Soporta rollback ante falla. Carga de observaciones y actualización de stock.

ProductoDAO

Listado de productos activos con soporte de filtrado por categoría. Actualización de stock al confirmar pedidos.

ReporteDAO

Consultas analíticas: ventas por mes (VentaPorMesDTO), ranking de productos (VentaPorProductoDTO) y resumen general (ResumenGeneralDTO).

Capa de Servicio

Patrón Factory: ServicioFactory centraliza la creación y el acceso a todas las instancias de servicio. Implementa double-checked locking con variables volatile para garantizar thread-safety cuando la GUI accede desde múltiples SwingWorker simultáneos. 
// ServicioFactory — Singleton Thread-Safe
private static volatile ProductoService productoService;

public static ProductoService getProductoService() {
    ProductoService result = productoService;
    if (result == null) {
        synchronized (LOCK) {
            result = productoService;
            if (result == null) {
                productoService = result = new ProductoService();
            }
        }
    }
    return result;
}

Conexión — HikariCP Pool

La clase ConexionDB fue migrada de DriverManager a un pool HikariCP que reduce la latencia de cada request de ~500ms a <5ms al reutilizar conexiones pre-establecidas.

Max Pool: 10 Min Idle: 2 Cache PrepStmts: 250 
-- Configuración de conexión (db.properties)
db.url=jdbc:mysql://<host>:4000/restaurant?useSSL=true
db.username=root
db.password=****
hikari.maximumPoolSize=10
hikari.minimumIdle=2
hikari.cachePrepStmts=true
hikari.prepStmtCacheSize=250
Diseño clave: La clase ConexionDB implementa un Singleton con fallback: si db.properties no está disponible (ej. entorno de testing), la conexión cae a valores hardcodeados de desarrollo local, sin romper los tests unitarios.

05

Optimizaciones Implementadas

Tras la integración inicial, se detectaron congelamientos severos de la UI. Se realizó una refactorización profunda que resolvió todos los problemas de rendimiento.

❌ Problema Original

Todas las consultas a la BD se ejecutaban en el EDT (Event Dispatch Thread), el único hilo de Swing. Como TiDB Cloud tiene latencia de red + TLS, cada getConnection() tardaba 500ms–5s, congelando completamente la pantalla.

✅ Solución Aplicada

Clase utilitaria AsyncDataLoader basada en SwingWorker. Todas las operaciones de BD se delegan a un hilo background. El EDT solo recibe el resultado final para actualizar la UI, nunca espera activamente.

// AsyncDataLoader — Operación asíncrona genérica
AsyncDataLoader.load(
    parentComponent,                     // Componente padre (se deshabilita durante la carga)
    () -> ServicioFactory.getProductoService().listar(),  // En hilo background
    productos -> actualizarTabla(productos),         // En EDT al finalizar
    error -> mostrarError(error.getMessage())           // En EDT si falla
);
Resultado: El login pasa de ~5 segundos a respuesta inmediata con cursor de espera. El cambio entre secciones es fluido. La UI nunca se "congela" independientemente de la latencia de red.

Conclusión

El proyecto logra una implementación completa y funcional de un sistema de gestión para restaurante, respetando los principios de arquitectura multicapa enseñados en la cursada. Los puntos más destacados son:

Separación de Responsabilidades

Cada capa (Modelo → DAO → Service → Controller → Vista) está claramente delimitada, con interfaces como contratos entre capas.

Patrones de Diseño Aplicados

DAO, Factory (ServicioFactory), Singleton (ConexionDB), Observer (AsyncDataLoader), Strategy implícita en los DTOs de reportes.

Calidad de Código

Tests unitarios automatizados con JUnit, uso de transacciones JDBC, manejo de excepciones, código libre de hardcodeo vía db.properties.

Proceso de Desarrollo

Git con ramificación estratégica (layer/dao, layer/service, layer/view, develop, master), integración continua por feature branches.
